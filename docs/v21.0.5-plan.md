## Hard architecture rules (unchanged)

1. Standalone, signal-based, OnPush.
2. SSR-safe. KaTeX and Mermaid both touch DOM — guard with
   `isPlatformBrowser` AND lazy-load via dynamic `import()`.
3. Tailwind v4 utility classes only.
4. `pk-` selector prefix, `Pk` class prefix, Component suffix.
5. Strict TypeScript. No `any`, no `@ts-ignore`.
6. Bun only.

## Spec

### New inputs on pk-markdown

- `enableMath: input<boolean>(false)`
- `enableDiagrams: input<boolean>(false)`
- `mathInlineDelimiter: input<'$' | '\\(' | 'both'>('$')`
- `mathBlockDelimiter: input<'$$' | '\\[' | 'both'>('$$')`

When both flags are false (default), `pk-markdown` behavior is
unchanged — same bundle, same dependencies, same output.

When `enableMath` is true:
- Lazy-load KaTeX in browser only via `await import('katex')`
- Detect math expressions in the markdown source per the configured
  delimiters
- Render inline math (`$x^2$`) and block math (`$$\int f(x)dx$$`)
- Default delimiters are dollar-sign style (`$` and `$$`) — most LLM
  outputs use these
- `'both'` option accepts either dollar-style OR LaTeX-style (`\(...\)`
  for inline, `\[...\]` for block)
- Use `katex/contrib/auto-render` for batch processing after the
  markdown HTML is rendered

When `enableDiagrams` is true:
- Lazy-load Mermaid in browser only via `await import('mermaid')`
- Detect ` ```mermaid ` fenced code blocks during markdown parsing
- Replace those code blocks with a Mermaid render container instead
  of the default `pk-code-block`
- Initialize Mermaid once with `securityLevel: 'strict'` (sandboxes
  the rendering, prevents script injection from model output)
- Render each diagram on a unique ID derived from the block content
- Failed diagrams (invalid syntax) fall back to rendering the source
  in a `pk-code-block` with `language="mermaid"` so the user sees
  what the LLM generated rather than a broken render

### ExtraDeps

Update markdown's `component-rule.ts` entry to include both deps
unconditionally:

- `marked: ^18.0.3` (already there)
- `katex: ^0.16.11` (new)
- `mermaid: ^11.4.0` (new)

Document loudly that katex and mermaid only load at runtime when
the corresponding input flag is true. The `extraDeps` install adds
them to `package.json` but the consumer's bundler will tree-shake
unused dynamic imports — KaTeX and Mermaid only enter the bundle
when their lazy-import code path actually runs.

This is the right tradeoff: simpler API, install-time deps, runtime
tree-shaking handles the bundle size question.

If a consumer audits their bundle and asks why `marked` is in their
`package.json` after `ng generate ngx-prompt-kit:markdown`, the same
answer applies — it's a runtime dep that may or may not load
depending on configuration.

The alternative (separate schematics like `markdown-math` and
`markdown-diagrams`) was considered and rejected as overcomplicating
the API for a marginal install-size win.

### Behavior details

#### Math rendering

After `marked` parses the markdown to HTML, run KaTeX's auto-render
on the resulting DOM. The integration:

- Dynamic-import `katex/contrib/auto-render`
- Call `renderMathInElement` with the host's content element ref
- Pass `delimiters: this.computedDelimiters()` (a computed signal
  returning the right delimiter array based on the two delimiter inputs)
- Pass `throwOnError: false` so bad LaTeX renders as the source text,
  not as an error

Math containers should inherit text color via Tailwind's `text-foreground`.
Override KaTeX's default styles minimally — only fix color inheritance
and font size, don't restyle the math itself.

#### Mermaid rendering

Two-phase.

**Phase 1 — override marked's code renderer for the `'mermaid'` language tag:**

- Build a custom `marked.Renderer()` that intercepts `code` calls where
  `lang === 'mermaid' && this.enableDiagrams()`
- Emit a placeholder `<div class="mermaid-container" data-mermaid-id="..." data-mermaid-source="...">SOURCE</div>`
- Use a stable hash (djb2 or similar) of the block source as the id
  so identical diagrams across re-renders share the same id and aren't
  re-rendered unnecessarily
- Always escape the source going into the data attribute and the
  fallback text content
- For non-mermaid code blocks, delegate to the existing
  `pk-code-block` rendering path

**Phase 2 — after the HTML is in the DOM, render each container:**

- Query all `.mermaid-container` elements
- Lazy-import mermaid: `const { default: mermaid } = await import('mermaid')`
- Initialize once: `mermaid.initialize({ securityLevel: 'strict', startOnLoad: false })`
- For each container, call `await mermaid.render(id, source)` to get the SVG
- Replace the container's contents with the SVG output

> **Security note for the SVG insertion step.** Replacing the container's
> contents with the rendered SVG means writing markup that came out of
> the Mermaid renderer back into the DOM. The protection here is layered:
> (1) `securityLevel: 'strict'` is non-negotiable — it sandboxes Mermaid's
> own rendering and disables click handlers, links, and HTML labels that
> LLMs sometimes generate; (2) the source string is hash-keyed and treated
> as opaque input to Mermaid's renderer rather than concatenated into a
> template; (3) on render failure (catch branch), fall back to a code
> block via `pk-code-block` with the source escaped — never inject the
> raw source as HTML. If a future Mermaid version introduces options that
> change the SVG output, audit again. Consider routing the final SVG
> through a sanitizer such as DOMPurify if the security posture needs to
> tolerate untrusted Mermaid versions or strict CSPs.

If `mermaid.render` throws (invalid Mermaid syntax — common from LLMs),
catch it and replace the container with a `pk-code-block` showing the
escaped source. The user sees what the LLM tried to produce, not a
silent failure.

#### Streaming behavior

`pk-markdown` handles streaming markdown (text arrives in chunks).
Math and diagram rendering should NOT fire on every chunk — that's
expensive and visually flickery. Two reasonable approaches:

- **Approach 1 (simpler):** debounce the rendering. Use a 300ms debounce
  on the rendering effect so partial math like `$x^` doesn't try to
  render until the closing `$` arrives.
- **Approach 2 (better but more complex):** only render math/diagrams
  when the streaming completes (signal goes from streaming=true to
  false). Requires a new `streaming` input on `pk-markdown`.

Go with **approach 1**. Simpler, doesn't require a new input, handles
both streaming and static cases. The 300ms feels instant for finished
content but lets in-progress content settle.

#### SSR safety

Both KaTeX and Mermaid call DOM APIs at render time. Wrap all
rendering work in `afterNextRender(() => { ... })` so it only fires
in the browser. The dynamic imports themselves are safe in SSR
(they just don't execute), but the rendering calls would crash.

## Anti-patterns

- Don't make math/diagrams default-on. The bundle cost is real;
  opt-in protects light consumers.
- Don't load KaTeX or Mermaid statically. Always dynamic `import()`.
- Don't re-render math/diagrams on every keystroke during streaming.
  Debounce.
- Don't override KaTeX's diagram-rendering CSS heavily. Minimal
  overrides only — color inheritance, font sizing.
- Don't fall back silently on Mermaid errors. Render the source as
  a regular code block so the user sees what the LLM tried to
  produce.
- Don't try to handle math + diagrams in the same code path. They
  are independent concerns; keep their implementations separate.
- Don't relax `securityLevel: 'strict'` on Mermaid — see the security
  note in the rendering section.

## Visual checks the human will be looking for

- Math renders inline within a paragraph without breaking line flow
- Block math is centered with appropriate top/bottom margin
- Mermaid diagram renders at sensible size — not stretched, not tiny
- Dark mode: math text is readable (KaTeX defaults to black; needs
  `color: inherit` fix)
- Dark mode: Mermaid diagrams are visible — Mermaid has dark theme
  support, set theme based on Spartan's resolved theme via
  `prefers-color-scheme` detection or a CSS variable check
- Bad math (`$\incomplete`) renders as source text, not as an error
- Bad Mermaid renders as a code block with the source visible

The dark mode for Mermaid is the most likely point of friction —
Mermaid's default theme is light. Look up the current Mermaid v11
API for `mermaid.initialize({ theme: 'dark' })` and detect Spartan's
active theme to switch dynamically. If detection is fragile, expose
a `mermaidTheme` input and let consumers control it.

## Two flags worth keeping in mind

- **Mermaid dark-mode detection** is the part most likely to cause
  friction. If Spartan exposes the resolved theme via a CSS variable
  or a service, use that — the `prefers-color-scheme` media query
  alone won't catch consumers who manually toggled the theme via
  Spartan's theme service. If the detection turns out fragile, the
  fallback (a `mermaidTheme` input) is fine — it just shifts the
  work to the consumer.
- **`securityLevel: 'strict'` on Mermaid is non-negotiable.** LLMs
  sometimes generate Mermaid that includes click handlers, links,
  or HTML labels — strict mode disables those. Without it,
  model-generated content could inject behavior.
