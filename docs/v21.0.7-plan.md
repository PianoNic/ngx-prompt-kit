## Overview

User-identity-integrated usage indicator. Three display modes
covering the real-world patterns:

- **ring** — circular progress ring wrapping the avatar
  (Instagram/Discord pattern — usage IS the chrome)
- **inline** — avatar + name + horizontal fill bar in a sidebar
  header row (less compact, exact numbers visible at a glance)
- **card** — standalone card with fill bar, label, count
  (JetBrains AI widget pattern — sidebar block, identity-optional)

Not redundant with existing components:
- `pk-token-counter` — input-side, char/token count as you type
- `pk-cost-display` — per-exchange running cost
- `pk-usage-card` — persistent session/account-level balance with
  visual fill state, identity-integrated

## API

```ts
pk-usage-card

Inputs:
  used: number (required)
  limit: number (required)
  display: 'ring' | 'inline' | 'card' (default 'ring')
  avatar: string (optional URL)
  name: string (optional)
  sublabel: string (optional — 'Pro plan' / 'Resets in 12d' /
    'Free tier')
  label: string (default 'Usage' — used in tooltip + a11y label)
  unit: string (default 'tokens')
  showPercentage: boolean (default true)
  locale: string (default browser)
  ringSize: number (default 40, ring mode only — avatar + ring
    diameter in px)
  ringStroke: number (default 2.5, ring mode only — stroke width)

Outputs:
  topUpClicked: () — only renders CTA when handler is wired
```

## Behavior per display mode

### ring

- SVG circle background (bg-muted stroke) + foreground stroke
  using stroke-dasharray for the progress arc
- Avatar sits inside the ring (img with rounded-full, sized to
  fit ringSize - 2*ringStroke - 2px gap)
- Ring stroke color uses threshold colors (muted → amber →
  destructive at 75/90/100%)
- When name set: avatar+ring on left, name + sublabel stacked
  to the right
- Tooltip on hover/focus: "used / limit · N%"
- Most compact mode

### inline

- Horizontal layout: avatar (if set) + flex-col with name +
  sublabel + thin fill bar (h-1.5)
- Bar uses same threshold colors
- Number readout below or beside the bar: "12,400 / 50,000"
- Designed for sidebar header rows where ring is too small for
  the numbers to be useful

### card

- hlm-card containing label at top, fill bar, used/limit count
- Avatar + name optional — when set, appears as small header
  above the bar; when omitted, just the usage data
- Top-up button at bottom-right when (topUpClicked) is wired
- Designed to be a discrete sidebar block

## Behavior shared across modes

- Threshold colors via traffic-light: 0–75% muted, 75–90% amber
  (text-amber-600 / dark:text-amber-500 / bg-amber-500), 90–100%
  destructive
- Numbers formatted via Intl.NumberFormat with the locale input
- Top-up button conditionally renders only when a (topUpClicked)
  handler exists — don't show a dead button
- When limit is 0 or null: degrade gracefully. ring shows full
  muted ring with no fill, inline hides the bar, card hides the
  bar. Just shows the used count + label.
- SSR-safe — pure presentational, no DOM access beyond standard
  Angular template rendering

## Demo (4 examples)

1. Ring mode with name + sublabel — the canonical "user card with
   ring" pattern. Use a placeholder avatar + name "Niclas",
   sublabel "Pro plan". 25% usage so the ring is mostly empty.
2. Inline mode — sidebar header row context. Avatar + name +
   sublabel "Resets in 12 days" + bar at 60%.
3. Card mode with top-up wired — full standalone card, 88% usage
   (amber threshold), Top-up button visible. Wire (topUpClicked)
   to a console.log so the button actually fires.
4. Threshold trio side-by-side using ring mode — three ringed
   avatars at 30% / 85% / 105% showing muted / amber /
   destructive ring colors. Most visually informative example.

## Reasons for deferral (recap)

- v21.0.6 already grew from 5 originals to 7 (model-list +
  model-browser added mid-session). Adding an 8th moves the
  release shape further from "agent-UI toolkit gaps" toward
  "kitchen sink."
- "1-2 hours of work" estimates have a poor track record this
  session — token-counter, message-edit, model-picker all grew
  past initial spec once in the file.
- Saving the spec to docs preserves the design while letting
  v21.0.6 ship and gather real-usage signal first.
